# Security

アカウントごとに習慣データを管理するためのセキュリティ方針です。

## 基本方針

- ユーザーごとにデータを分離する
- 誰が読めるか、誰が書けるかを必ずサーバー側で確認する
- 暗号化だけに頼らず、認証と認可を先に設計する
- 機密情報は必要以上に保存しない
- パスワードを保存する場合は平文保存しない
- `.env` や秘密情報はAIに読ませない

## 秘密情報

- `.env` はコミットしない
- `.env.example` にはキー名だけを書き、実値を書かない
- APIキー、DBパスワード、トークン、秘密鍵をチャットに貼らない
- AI に秘密情報の中身を確認させない
- 秘密情報が必要な作業では、キー名、用途、設定場所だけを共有する
- 誤って共有した秘密情報は漏洩した前提でローテーションする
- サーバーログ、エラーログ、監査ログに秘密情報を出さない

## 認証

- アカウントごとにログインできるようにする
- 認証は Cognito に委譲する
- v1 Web ではメール/パスワード、Google、Apple をまとめて有効化する
- アプリDBは Cognito の `sub` をユーザー特定の基準にし、自前のパスワードハッシュは持たない
- セッション、トークン、リフレッシュトークンは安全に保存する
- ログイン失敗時は、ユーザーが存在するか分からないエラー文にする

## 認可

- すべてのユーザーデータに `userId` を持たせる
- 読み込み、作成、更新、削除のたびに `userId` を確認する
- クライアントから渡された `userId` を信用しない
- サーバー側で認証済みユーザーを特定し、そのユーザーのデータだけ操作する
- 管理者機能を作る場合は通常ユーザーと権限を分ける

## データ

| データ | 扱い |
| --- | --- |
| 習慣 | ユーザーごとに分離する |
| 達成記録 | ユーザーごとに分離する |
| 振り返り | 私的な内容になりやすいので慎重に扱う |
| 通知設定 | ユーザーごとに分離する |
| パスワード | 平文保存しない |
| セッション / トークン | 漏洩前提で短命化や失効を考える |

## 暗号化

- 通信は HTTPS/TLS を前提にする
- サーバー保存する機密データは暗号化 at rest を検討する
- パスワードは暗号化ではなくハッシュ化する。アプリDBでは保持せず、Cognito に委譲する
- 振り返りなど特に私的な本文は、将来的にユーザー単位の暗号化を検討する
- エンドツーエンド暗号化は必要性が明確になってから検討する

## 保存

- Free の保存はスマホ端末内データとして扱う
- サーバー同期する段階では、Repository 経由で保存する
- DB設計は `docs/db-design.md` を基準にする
- 保存データにはバージョンを持たせる
- 古い保存データはマイグレーションする
- 削除したユーザーのデータを残すかどうかを決める

## API

- API は認証必須にする
- 自分のデータだけ取得・更新できるようにする
- ID指定の API は所有者チェックを必ず行う
- 入力値はサーバー側でも検証する
- エラーに内部情報を出さない

## ログ

- パスワード、トークン、認証コードをログに出さない
- 振り返り本文など私的な内容を不用意にログに出さない
- 認証失敗、権限エラー、重要な削除は監査ログ候補にする

## 最初に決めること

- ユーザー削除時のデータ削除方針
- 振り返り本文を通常保存するか、追加暗号化するか

## 実装前チェック

- 認証済みユーザーをサーバー側で特定できるか
- すべてのユーザーデータに所有者があるか
- 他ユーザーのIDを指定しても読めないか
- 他ユーザーのIDを指定しても更新・削除できないか
- パスワードやトークンがログに出ないか
- 保存データの復元とマイグレーションがあるか

## セキュリティテスト

- 未ログインではユーザーデータ API を使えないこと
- 他ユーザーのデータを読めないこと
- 他ユーザーのデータを更新・削除できないこと
- クライアントから別の `userId` を送っても無視されること
- パスワードが平文保存されないこと
- 認証失敗時にユーザー存在有無が分からないこと
- トークンや認証情報がログに出ないこと
- 破損した保存データでもアプリが落ちないこと
